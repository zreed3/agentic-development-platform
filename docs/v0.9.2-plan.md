# ADG v0.9.2 Plan

Builds on the v0.9.1 baseline ([`field-report-and-hardening-plan.md`](field-report-and-hardening-plan.md)).
**Holds for review before development**, per the v0.9.1 convention. No code in
this change — refined brief + plan only.

## How this plan was produced (ADG dogfood)

The raw request was refined with the platform's own tooling rather than by eye:

- **`work:classify`** was run on each distinct intent to assign a Proofline lane
  and required gate (table below — these are tool outputs, not guesses).
- The **elicitation discipline** (brief → discrete items → surfaced gaps →
  non-goals) was applied to turn five run-on asks into six bounded work items.
- The plan is anchored to the **existing** plugin/install architecture
  (`plugins/adg-governance/`, `scripts/adg-install.mjs`) so it extends ADG instead
  of bolting something parallel on.

First implementation step (M1.0) is to seed these six items into the SQL backlog
and run `elicitation:graph` / `ux:validate` on them — so ADG governs its own
v0.9.2 the way it governs everything else.

## Refined brief (the improved request)

> **Intent.** Make ADG's governance *enforceable inside the harness that's
> actually doing the work* (Claude Code first), keep it *trivially updatable*, and
> close the v0.9.1 evidence-rigor gaps — then make the platform *teachable* via a
> public site.

Five asks in the original prompt → six bounded work items:

| Original ask | Becomes |
|---|---|
| "Implement recommended changes" | **F-A** evidence tiers + **F-B** adopter doctor (the v0.9.1 R1/R2) |
| "Claude Code plugin to make it more deterministic" | **F-C** Claude Code plugin (hooks + permissions + commands + skills) |
| "CLAUDE.md that matches AGENTS.md" | **F-D** CLAUDE.md as a *generated mirror* of AGENTS.md |
| "Packaged, tightly connected to the dev harnesses + easily updated" | **F-E** package as a Claude Code plugin *marketplace* + extend `adg:install`/`adg:update` |
| "Simple cPanel site to educate people" | **F-F** static education site, deployable to cPanel |

## Work items, lane-classified

Lanes are `work:classify` output for each item:

| Item | Lane | Full gate? | Why |
|---|---|---|---|
| **F-A** Evidence tiers + release rule | **L4 release-signoff** | yes | schema + release-gate semantics |
| **F-B** Adopter conformance doctor | **L3 sensitive** | on schema/guardrail change | governance tooling |
| **F-C** Claude Code plugin | **L3 sensitive** | on policy/hook change | enforces guardrails in-harness |
| **F-D** CLAUDE.md generated mirror | **L3 sensitive** | on rulebook change | touches AGENTS.md |
| **F-E** Packaging + install/update | **L3 sensitive** | on installer change | mutates host repos |
| **F-F** cPanel education site | **L2 bounded slice** | no | docs/marketing, no policy |

## Milestones & sequence

### M1 — Evidence hardening (v0.9.1 R1/R2)
- **F-A · Evidence tiers (L4).** Add ordered `evidence_tier` enum
  (`asserted` < `config` < `test` < `live`) to the audit/verify schema + seed;
  thread through `backlog:verify` and `audit:record`; add a release-gate view
  predicate so declared-sensitive classes (deploy/infra/perf/runtime-sec/residency)
  cannot be signed off on `config`/`test` alone. Files: `data/schema.sql`,
  `scripts/backlog-db.mjs`, `scripts/record-audit.mjs`, `scripts/validate-audit.mjs`,
  `AGENTS.md`, `docs/sql-data-layer.md`. Gate: full governance; test that a
  sensitive claim with only `config` evidence fails the gate and a `live` event
  clears it.
- **F-B · Adopter doctor (L3).** `scripts/adg-doctor.mjs` + `adg:doctor`: fail when
  an install tracks generated `*.sqlite`/mirrors, bolts on a committed-artifact
  diff gate, or embeds git SHA/timestamps in tracked docs. Wire into
  `adg:install:status` and `adg:update` warnings. Test: fixture install with a
  tracked `*.sqlite` fails doctor.

### M2 — Claude Code integration (the new core ask)
- **F-D · CLAUDE.md as a generated mirror (L3).** A `scripts/adg-claude-md.mjs`
  generator emits `CLAUDE.md` from `AGENTS.md` (single source of truth; CLAUDE.md
  carries a "generated — edit AGENTS.md" banner). Doctor (F-B) gains a check that
  the two are in sync — so the rulebook can never silently diverge across harnesses
  (the F1 anti-drift principle, applied to itself).
- **F-C · Claude Code plugin (L3).** A sibling `plugins/adg-governance/.claude-plugin/`
  to the existing `.codex-plugin/`, sharing the same `skills/` and neutral manifest
  (bump plugin to `0.4.0`). **Determinism is delivered by four deterministic
  surfaces** (see design below): hooks, a permissions template, slash commands, and
  CLAUDE.md. Gate: plugin:validate + hook smoke test (a forbidden bulk-file read is
  blocked; a destructive command is denied).
- **F-E · Packaging + easy updates (L3).** (a) Add a top-level
  `.claude-plugin/marketplace.json` so the repo is installable with
  `claude plugin marketplace add <repo>` / `claude plugin install adg-governance`
  — the "tightly connected to the harness" mechanism. (b) Extend
  `scripts/adg-install.mjs` with a `--client claude` path that installs/refreshes
  the plugin, generated `CLAUDE.md`, hooks, and a `.claude/settings.json` template
  into a host repo, tracked in `adg-install-state.json` with timestamped backups —
  so `adg:update` refreshes them without chat instructions.

### M3 — Education
- **F-F · cPanel education site (L2).** A static, dependency-free site (extends the
  existing `index.html` / `docs/*.html`) covering: what ADG is, the context
  inversion, the lanes, the SQL data layer, and install — deployable by upload to
  cPanel (no build step, relative asset paths). Optional `npm run site:build` that
  assembles `dist-site/` from the docs.

### M4 — Downstream
- **bord.room remediation (v0.9.1 R3).** Run the new `adg:doctor` against
  bord.room; gitignore its generated `*.sqlite`/mirrors, de-fang the
  committed-artifact drift gate, stop embedding git SHA in tracked docs. Separate
  change in that repo, now driven by tooling rather than this prose.

## Determinism design — how the plugin makes Claude Code deterministic

Claude Code's deterministic levers, each mapped to an ADG control:

1. **Hooks** (`hooks/hooks.json`) — the core lever. A `PreToolUse` hook routes
   risky tool calls through `guardrail-check.mjs` (deny-by-default by risk class);
   a read/grep hook blocks `forbiddenBulkFiles` (the context hazards); an optional
   `Stop` hook runs the lane's gate. Hooks run deterministically on every matching
   event — not "please remember to," but enforced.
2. **Permissions template** (`.claude/settings.json`) — allow/deny/ask lists that
   mirror the guardrail risk classes (read-only/code-change allowed; migration/
   secrets/production ask; destructive denied).
3. **CLAUDE.md** (F-D) — the always-loaded rulebook, generated from AGENTS.md.
4. **Slash commands** (`commands/*.md`) — `/adg-classify`, `/adg-context`,
   `/adg-verify` (evidence-tier aware), `/adg-doctor` — repeatable wrappers over
   the existing scripts, so the workflow is canned, not improvised.

## Decisions needed before development

1. **CLAUDE.md form** — generated mirror of AGENTS.md (recommended, no drift) vs a
   thin pointer (`"Read AGENTS.md"`) vs a hand-maintained full copy. *Recommend:
   generated.*
2. **Hook aggressiveness** — block (deny) vs warn for the *sensitive* classes.
   *Recommend: deny destructive/forbidden-bulk-file; ask on migration/secrets/
   production; warn on sensitive.* Hooks must fail safe (a hook error blocks, not
   allows).
3. **Distribution** — marketplace + copy-install both (recommended), or only the
   existing copy-install? Marketplace is what makes it "tightly connected to the
   harness."
4. **cPanel site scope** — single landing page vs a small multi-page site; which
   domain; static-only confirmed (no server-side).
5. **Versioning** — land all of M1–M3 as **v0.9.2**, or ship M1 as v0.9.2 and the
   Claude integration as v0.9.3? *Recommend: M1 = v0.9.2, M2 = v0.9.3, so the
   evidence hardening isn't gated on the larger plugin work.*

## Non-goals (inherited from the roadmap)

- No general-purpose agent framework; no autonomous runtime in the plugin (it
  *constrains* the harness, it doesn't replace it).
- No vector search; no SaaS dependency.
- The plugin does not weaken deny-by-default or append-only audit for convenience.
- CLAUDE.md and AGENTS.md stay one source of truth — never two hand-edited copies.
