SQL-first backlog

SQLite selects the context. SQLite is queried, never pasted.

The platform's "SQL server" is a single local SQLite database — no daemon, no network, no driver. It is rebuilt from tracked text sources, queried by the context broker, and treated as generated rather than canonical.

Runtime requirements 2 Node ≥20 and the sqlite3 CLI. No npm install, no database server, no daemon.
Overview / SQL Data Layer

Canonical vs generated

The core split

A deliberate split keeps the database cheap to trust and easy to review. There are three layers, and knowing which is which matters.

Layer Files Role Version-controlled?
Canonical, human-editable data/seed/backlog.seed.json, data/audit/audit-log.jsonl The source of truth you edit. Yes — always committed
Generated, queryable data/backlog.sqlite Rebuilt from the canonical sources. What you query. No — gitignored, rebuilt by npm run setup
Generated, reviewable data/backlog-source.sql (.dump), data/schema.sql (.schema) Text mirrors for diffs and code review. Yes — committed as review artifacts

Key rule: data/backlog.sqlite is not version-controlled. Neither are any other generated *.sqlite databases. If an adopter starts committing these files, the adg:doctor conformance check will flag it as a violation — it re-introduces exactly the anti-pattern ADG was built to prevent.

The item lifecycle

Claim model

Items move through a claim/transition model recorded as immutable events. The current status is computed from those events — the history is never overwritten.

1 backlog:next
2 claim
3 start
4 complete
5 verify 
npm run backlog:next   -- --feature S07 --type task --limit 5
npm run backlog:show   -- --item S07-TASK-01
npm run backlog:claim  -- --item S07-TASK-01 --actor agent --ttl-hours 8 --scope scripts/agent-context.mjs
npm run backlog:start  -- --item S07-TASK-01 --summary "Started"
npm run backlog:complete -- --item S07-TASK-01 --summary "Implemented" --evidence scripts/agent-context.mjs
npm run backlog:verify -- --item S07-TASK-01 --summary "Verified" --evidence "npm run test:agent-context"
npm run backlog:release -- --item S07-TASK-01    # give up a claim
npm run backlog:active                            # list unexpired claims

Claims have a TTL and a write scope. backlog:next skips items with an active claim, so parallel agents can each take disjoint work without colliding — the database is the coordination point.

Append-only by construction

Event sourcing

The audit log is the canonical source

data/audit/audit-log.jsonl is append-only. Never delete or rewrite an event. If a past one was wrong, append a corrective comment or decision event.

This means neither the agent nor a future you can quietly launder history.

Current state is always derived

A feature's current status, an item's lifecycle state, an active claim — all are computed by SQL views over the event stream. Nothing is ever edited in place.

That makes the audit trail a real trust primitive instead of decoration.

Event types in the audit log

Event typeUse for
statusRecording a feature or item status change.
evidenceAttaching verified evidence to a feature or item.
test-resultRecording a gate or check outcome (pass or fail).
decisionWaiving a gate — must include reason, risk, and rollback.
commentA corrective note appended to an earlier event.

Querying directly

Shell examples

Everything is plain SQL. Run these after npm run setup:

# Feature status board
sqlite3 data/backlog.sqlite \
  "select feature_id,planned_status,current_status,latest_update from feature_current_status order by feature_id;"

# What is claimed right now
sqlite3 data/backlog.sqlite \
  "select item_id,actor,expires_at from backlog_item_claims where status='active';"

# Audit trail for a feature
sqlite3 data/backlog.sqlite \
  "select occurred_at,event_type,status,summary from audit_events where feature_id='S07' order by occurred_at;"

Adopting it in a new project

Host repo setup

The schema is generic — nothing in it is specific to any one product. To adopt the layer in a host repo:

  1. 1
    Replace the seed

    Edit data/seed/backlog.seed.json with your project's epics, features, items, routes, and personas.

  2. 2
    Rebuild and validate

    npm run setup && npm run backlog:validate. The validate command enforces: every feature has at least one task and one test case, no orphaned dependencies or routes.

  3. 3
    Keep the .sqlite on .gitignore

    The database is generated from the text sources. Committing it re-introduces the anti-pattern ADG was built to prevent.
