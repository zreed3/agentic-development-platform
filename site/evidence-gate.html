Evidence tiers & release gate

Config-existence is not proof. The gate can tell the difference.

ADG's evidence model has four ordered tiers. Sensitive release classes — deploy, infra, performance, data-residency, runtime-security — cannot be signed off on config or test alone. They require a live event.

Evidence tiers 4 asserted < config < test < live. Only live evidence closes a sensitive release claim.
Overview / Evidence Tiers & Release Gate

Why evidence tiers exist

Motivation

This problem emerged from real first-party use of ADG on a production system. A deployment task was recorded verified based solely on a Terraform config declaring the correct region. It was wrong.

The concrete failure: A task to pin serverless functions to the Sydney region (co-located with an Australian database) was verified on the strength of the config declaring syd1. Production HAR captures later proved the functions were executing in iad1 (US East) while the database sat in Sydney — every query crossing the Pacific, producing 4–6 second pages. The audit trail looked complete and honest; the claim was simply false at the only layer that mattered.

The backlog lifecycle (claim → … → verify) recorded evidence as a free-form command or path, with no distinction between "the configuration that should make this true exists" and "this was observed true in the running system." Evidence tiers close that gap.

The four evidence tiers

Ordered scale
asserted Lowest
config Configuration exists
test Automated check passed
live Highest — observed true
Tier Meaning Examples
asserted A human or agent claim with no supporting artifact. "I believe this is done." A prose note in an audit event.
config The controlling configuration exists — Terraform, environment variable, feature flag. Terraform plan shows syd1. A .env setting is present. A flag is enabled in config.
test An automated check passed — unit test, integration test, governance gate. npm run ci:governance passed. A test suite is green. A contract test validates the API shape.
live Observed true in the running or deployed system. A probe confirms the region. A response header shows the correct datacenter. A measured latency metric. A restore drill succeeded.

The release gate rule

When live is required

Claims in declared-sensitive classes cannot gate a release on config or test alone. They require a live evidence event.

This is a few-line addition to the schema — an enum column plus one view predicate on the release-gate view. It directly prevents the class of false-confidence failure described above.

Sensitive release classes requiring live evidence

deploy

Any claim that a deployment completed successfully in a real environment.

infra

Infrastructure configuration that affects availability, region, or capacity.

performance

Performance targets or SLAs (latency, throughput, page load time).

runtime-security

Security controls active in the running system (TLS, headers, firewall rules).

data-residency

Data locality or sovereignty claims (region, jurisdiction, transfer restrictions).

Recording evidence in the audit log

Commands

Evidence is attached to backlog items and audit events. Always be specific — a vague note is asserted tier at best.

# Record a verified item with explicit evidence tier
npm run backlog:verify -- --item S07-TASK-01 \
  --summary "Context broker smoke test passes" \
  --evidence "npm run test:agent-context" \
  --evidence-tier test

# Record a live-tier event in the audit log
npm run audit:record -- \
  --feature S07 \
  --type evidence \
  --status verified \
  --summary "Confirmed broker returns bounded packets in production-equivalent run" \
  --evidence "npm run test:agent-context" \
  --evidence-tier live

# A deployment claim with only config-tier evidence — will FAIL the release gate
# for sensitive classes until a live event is also recorded
npm run backlog:verify -- --item S04-TASK-01 \
  --summary "Sydney region configured" \
  --evidence "terraform plan" \
  --evidence-tier config

Rule: Never record failed gates as passing. If a check fails, record it as a failure event — do not summarize it away. The audit trail is only useful if it is honest about failures.

The full governance gate

ci:governance

The full gate is required at L4 and at any checkpoint where a policy, schema, or governance-behavior change has occurred.

npm run ci:governance

The gate runs all of these in sequence, and fails at the first error:

Command Checks Required when
backlog:validateEvery feature has tasks and test cases; no orphaned items.After any backlog changes.
audit:validateAudit log is parseable, IDs are unique, no obvious secrets.After any audit append.
guardrails:checkPolicy is valid, deny-by-default is set, risk classes are consistent.When tool/action policy changes.
agent:evalsAll eval scenarios pass their expected security outcomes.When guardrail policy or AI-security behavior changes.
metrics:doraDORA-style delivery metrics can be derived from git + audit.For delivery-process changes.
test:agent-contextContext broker emits valid, bounded packets; forbidden files are excluded.Any context-broker or profile change.

Waived gates need an audit decision. If a gate must be skipped, record a decision event in the audit log with reason, risk, and rollback. Do not weaken the policy to make work proceed.
