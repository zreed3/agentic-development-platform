Proofline v0.9

Cheap work stays cheap. Every serious claim has a proof line.

Proofline is ADG's delivery discipline. It separates exploration from claims by classifying every task into a lane before work begins. L0 spikes cost nothing; L4 release signoff requires the full gate.

Delivery lanes 5 L0 spike through L4 release signoff. The gate matches the risk — no more, no less.
Overview / Proofline Delivery Lanes

Classify first

The discipline

The goal is not to run full governance on every small change. The goal is to separate exploration from claims.

Step 1 Classify
Step 2 Choose lane
Step 3 Bounded context
Step 4 Lane-sized checks
Step 5 Upgrade if risk appears

The classifier is deterministic and conservative. If local evidence raises risk, upgrade the lane immediately. The lane cannot be downgraded once sensitive work is discovered — only upgraded.

The five lanes

Reference
L0

Spike

Use for: Read-only exploration, options, triage, debugging.

Evidence required: Notes only. No implementation or signoff claim.

L0 can say: "observed", "checked". It cannot say "verified", "release-ready", or "signed-off" — those words require L4.

L1

Quick-fix

Use for: Copy, CSS, small UI state, docs, obvious low-risk bugs.

Evidence required: Nearest check, screenshot, or smoke proof.

L1 can say: "changed", "observed". Full governance gate is not required.

L2

Bounded slice

Use for: Normal scoped implementation — a feature slice with clear boundaries.

Evidence required: ADG context slice plus targeted tests.

Example: This education site (F-F) is classified L2 — docs/marketing, no policy change.

L3

Sensitive

Use for: Auth, RBAC, tenant/business scope, schema, migrations, secrets, billing, production, guardrails, audit, governance tooling, CI policy.

Evidence required: Policy-specific gate, negative tests where applicable, and ADG validation.

L4

Release signoff

Use for: GitHub push/PR, pre-push, release candidate, GA, release, verified, release-ready, or signed-off claims.

Evidence required: Full governance / traceability gate (npm run ci:governance). GitHub-bound changes must always pass through here.

npm run adg:guard -- --event github-push --intent "GitHub update pre-push"
npm run ci:governance

The signoff boundary

What "done" means

L0 and L1 can say

  • "observed"
  • "changed"
  • "checked"

They cannot say "verified", "release-ready", or "signed-off". Those words require L4.

L4 can say

  • "verified"
  • "release-ready"
  • "signed-off"

Only after the full governance gate has passed. npm run ci:governance must succeed before any of these claims are recorded in the audit log.

Output discipline (L0/L1)

Low-overhead format

For L0 and L1, keep agent output short. Do not paste large context packets, generated mirrors, or full test logs unless the failure needs them.

lane L1 quick-fix
files docs/setup.html
checks screenshot
full gate no
next done

Classification commands

Usage 
# Classify a task before starting
npm run work:classify -- --intent "quick css spacing fix" --file docs/setup.html
npm run work:classify -- --intent "change tenant permission checks" --file src/auth.ts --format json

# Gate before a GitHub push
npm run adg:guard -- --event github-push --intent "GitHub update pre-push"
npm run ci:governance

# Install Proofline into a host repo
npm run adg:install -- --target /path/to/host-repo

# Check install state
npm run adg:install:status -- --target /path/to/host-repo --format json
