Overview / Context Inversion

The core inversion

Philosophy

Mainstream context tools (Repomix, code2prompt, files-to-prompt) maximize what they pack in. This platform does the opposite.

The intuitive (wrong) approach

Feed the agent the full tracker JSON, the SQL dump, and the generated HTML dashboard — "so it is well-informed." This approach pastes hundreds of thousands to millions of tokens into a context window before the agent writes a single line.

The ADG approach

Those exact files go on a forbiddenBulkFiles denylist. SQLite is positioned as the thing that selects a capped packet of context — feature, items, routes, recent audit events, and the specific files to read next.

Key insight: Your richest artifacts are the ones most likely to poison the context window. The broker's whole job is to withhold them and hand back only pointers. Aider's repo map is the closest cousin — but it ranks code symbols; ADG selects by project intent and traceability (feature → items → routes → anchors → audit).

The bounded context pipeline

How it works

Every context request follows this six-step flow:

Step 1 Task
Step 2 Classify
Step 3 SQL lookup
Step 4 Capped packet
Step 5 Anchored files
Step 6 Targeted checks

The agent reads only the files the packet names. A route packet is not allowed to balloon into the whole repo.

Four enforcement mechanisms

Implementation

1. Forbidden bulk files

Each context profile inherits a forbiddenBulkFiles list. The broker will never name these in a packet. npm run context:audit fails if a packet manifest references one.

  • data/backlog-source.sql
  • data/backlog.json
  • data/audit/audit-log.jsonl
  • docs/generated/**

2. Per-workflow caps

A profile caps audit rows, backlog events, routes, and — most importantly — maxFiles. Defaults are conservative:

  • Max audit events: 5
  • Max backlog events: 5
  • Max backlog items: 12
  • Max routes: 12
  • Max files: 8–12 (varies by workflow)

3. Query selection

A profile lists which SQL queries run. A docs task does not pull route files; an rbac task pulls persona workflows and negative-test anchors first. Available queries include:

  • feature_current_status
  • backlog_items_by_feature
  • routes_by_feature
  • persona_workflows_by_feature
  • recent_audit

4. TOON transport

For uniform arrays (routes, items, audit rows) the broker can emit TOON: a compact, header-plus-rows table format. JSON/SQL/SQLite stay canonical; TOON is a transport used only for the LLM-facing packet, and only after measuring it is actually smaller.

Measured sizes — demo backlog

Evidence

Generated with the seeded demo backlog (7 features, 57 items, 8 audit events):

Artifact Size Note
context:feature S07 route (toon) ~2.0 KB LLM-facing packet — this is what the agent receives
context:feature S07 route (markdown) ~2.8 KB Human-facing packet
context:feature S07 route (json) ~7.5 KB Machine assertions
data/backlog-source.sql ~26 KB Forbidden bulk file — on the denylist
data/backlog.sqlite ~164 KB Queried, never pasted into context

The packet is roughly an order of magnitude smaller than the SQL dump and ~80x smaller than the database — on a tiny demo backlog. The gap widens sharply with scale. In the product repo this layer was extracted from, the generated tracker JSON and SQL dump were multiple megabytes each: hundreds of thousands to millions of tokens.

Commands

Usage 
# Generate a bounded context packet for a feature
npm run context:feature -- --feature S07 --workflow route

# Item-scoped packet in TOON format (most compact)
npm run context:item -- --item S07-TASK-01 --workflow route --format toon

# Audit the last packet manifest for violations
npm run context:audit

# Reproduce size measurements
npm run setup
for fmt in markdown json toon; do
  printf "%-9s " "$fmt"
  node scripts/agent-context.mjs feature --feature S07 --workflow route --format "$fmt" --no-manifest | wc -c
done
