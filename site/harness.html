Dual-harness model

Codex and Claude Code. One policy.

ADG is published as one plugin with two adapters so Codex and Claude Code can run in the same repo and follow the same rules. The conformance doctor catches when an install has silently drifted from those rules.

One source of rules, two adapters

Architecture

Because CLAUDE.md is generated from AGENTS.md, the two harnesses cannot read different rulebooks — that is the structural guarantee of "same rules."

Shared (single source of truth) Used by
config/agentic/guardrails.json — deny-by-default risk classes Both harnesses
AGENTS.md rulebook → generated CLAUDE.md (npm run claude:generate) Both (identical text, structurally enforced)
plugins/adg-governance/agentic-plugin.manifest.json — neutral commands + risk classes Both harnesses
plugins/adg-governance/skills/ — portable skills Both harnesses
plugins/adg-governance/hooks/adg-guardrail-hook.mjs — the enforcement script Claude Code (via hook); Codex via adapter when available

Codex adapter

.codex-plugin/plugin.json — skills + commands + interface. Enforces through its approval policy plus the shared guardrails.json and the skills/commands that carry the same risk classes.

When Codex exposes a pre-tool hook surface, the same adg-guardrail-hook.mjs can be wired behind an adapter — it reads a neutral {tool_name, tool_input} JSON event on stdin.

Claude Code adapter

.claude-plugin/plugin.json — skills + commands + the PreToolUse hook. Enforces deterministically: the hook hard-blocks destructive commands and raw reads of generated context hazards (exit 2), and asks on secrets / production / migration.

The rules become physically enforced at the tool gate, not just documented.

Four deterministic surfaces

How Claude Code becomes deterministic

Claude Code's determinism is delivered by four surfaces, each mapped to an ADG control:

  1. 1
    Hooks (hooks/hooks.json) — the core lever

    A PreToolUse hook routes risky tool calls through guardrail-check.mjs (deny-by-default by risk class). A read/grep hook blocks forbiddenBulkFiles. An optional Stop hook runs the lane's gate. Hooks run deterministically on every matching event — not "please remember to," but enforced.

  2. 2
    Permissions template (.claude/settings.json)

    Allow/deny/ask lists that mirror the guardrail risk classes: read-only and code-change are allowed; migration, secrets, and production ask; destructive is denied.

  3. 3
    CLAUDE.md — the always-loaded rulebook

    Generated from AGENTS.md by npm run claude:generate. The doctor fails if the two files drift. One source of truth, two harnesses, zero manual sync.

  4. 4
    Slash commands

    /adg-classify, /adg-context, /adg-verify (evidence-tier aware), /adg-doctor — repeatable wrappers over the existing scripts, so the workflow is canned, not improvised.

The conformance doctor (adg:doctor)

Drift detection

ADG's rule is explicit: generated *.sqlite databases are gitignored, not committed. An adopter can silently re-introduce the exact anti-pattern ADG was built to avoid — and until the doctor was added, nothing caught it.

From a real session: Recording one fact in an adopter repo became a four-step dance — append audit event → regenerate → commit → drift gate goes red on data/backlog.* (which lag one regeneration pass) → git commit --amend → drift green. Hit twice in one session for single low-risk events. The doctor is the fix.

What the doctor checks

Generated files tracked in git

Generated *.sqlite or JSON/SQL mirrors are git-tracked. These should be on .gitignore.

A 13–23 MB binary .sqlite change per governance commit is a clear sign the repo has drifted.

Committed-artifact diff gate

A custom gate diffs committed generated artifacts against a fresh regeneration. This re-introduces the regenerate-then-amend tax that makes cheap work expensive.

Volatile provenance in tracked docs

Generated docs embed the git SHA or a timestamp in tracked frontmatter. These churn on every commit, creating unreviewable diffs.

AGENTS.md / CLAUDE.md sync

The two rulebook files must be identical. The doctor fails if they have drifted, preventing silent divergence between harnesses.

# Run the conformance doctor
npm run adg:doctor

# Check install state (includes doctor findings)
npm run adg:install:status -- --target /path/to/host-repo --format json

Using both harnesses in one repo

Setup
  1. 1
    Install the marketplace (Claude Code)

    claude plugin marketplace add <repo-url> then claude plugin install adg-governance.

  2. 2
    Install or refresh the Codex adapter

    npm run adg:install -- --target <repo> copies the portable lane policy, classifier/guard, and docs into the host repo.

  3. 3
    Generate and check CLAUDE.md

    npm run claude:generate mirrors AGENTS.md → CLAUDE.md. npm run claude:check fails if they drift. Wire claude:check into the governance gate.

  4. 4
    Both adapters now share one policy

    Every action resolves against the one guardrails.json. There is no conflict running both: they read per-harness rulebook files that are guaranteed identical.

No conflict between harnesses. Codex reads AGENTS.md; Claude Code reads the generated CLAUDE.md. Both are identical by construction. Only the Claude adapter registers a hook; Codex follows the same rules through its approval policy.
